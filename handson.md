## 2. AI 동료가 회사 데이터에 닿는 원리 (MCP) (20분)

> 방금 써 본 **NotebookLM · Deep Research · Idea Generation**은 **Google이 만든** 에이전트였습니다. 이런 에이전트가 **우리 회사 데이터·시스템**에 닿는 원리가 바로 **MCP**입니다. 이제 그 원리를 보고, 직접 **No-Code로 내 도구를 묶은 에이전트**를 만듭니다.

**한 줄 개념**

- **두뇌 = LLM** : 생각·판단
- **팔다리 = Tool** : 행동·정보 수집
- **신경 = MCP** : 두뇌가 팔다리를 호출하는 표준 통신 규약 (USB-C 같은 표준)

### 에이전트 vs MCP — 헷갈리지 않기

한 줄로: **에이전트는 '일하는 주체(수행비서)', MCP는 그 비서가 도구에 연결되는 '표준 규격'**입니다. 둘은 경쟁이 아니라 **포함 관계** — 에이전트가 MCP를 *씁니다*.

- **에이전트(Agent)** = 두뇌(LLM) + 지시(역할·목표) + 골라 묶은 도구 + 기억 + 실행 루프(PLAN→DELEGATE→SYNTHESIZE→RESPOND). 무엇을 할지 스스로 정하고 도구를 골라 실행하는 **'몸 전체'**.
- **MCP** = 그 에이전트가 **도구에 닿는 표준 연결**. 비유의 **'신경'**. USB-C처럼 어떤 도구든 같은 방식으로 꽂힙니다.

| 구분 | 에이전트(Agent) | MCP |
|---|---|---|
| 정체 | 일하는 주체 — 수행비서 | 도구 연결 규약 — 표준 케이블 |
| 범위 | LLM + 지시 + 도구 + 기억 + 루프 (**전체**) | 에이전트와 도구 사이의 **한 계층** |
| 비유 | 비서라는 '사람' 전체 | 비서가 전화·시스템에 꽂는 표준 단자 |
| 개수 | 한 에이전트가 **여러** MCP 도구 사용 | 한 MCP 도구를 **여러** 에이전트가 공유 |
| 핵심 질문 | "**누가** 일하는가" | "도구에 **어떻게** 연결되는가" |

> 헷갈림 방지 — **MCP를 붙였다고 에이전트가 되는 게 아닙니다.** 에이전트는 LLM+지시+도구+루프 *전체*이고, MCP는 그중 **도구 연결** 부분의 표준입니다.

### 오늘의 도구 카탈로그 — 8개 블록을 어디에 끼울까

> **머릿속 그림**: 에이전트(LLM) 가운데에 두고, 주변에 도구 8개가 USB 슬롯처럼 꽂혀 있습니다. 각 에이전트는 **필요한 것만** 골라 끼워요.

```
                    ┌────────────────────────────┐
   Gmail ────┐      │                            │      ┌─── 임직원 디렉토리 MCP
   Drive ────┼──────│       에이전트 (LLM)        │──────┼─── 거래/고객 DB MCP
   Calendar ─┘      │   PLAN→DELEGATE→SYNTH→RESP │      └─── 외부 회사 검색 MCP
                    │                            │
                    └──────────────┬─────────────┘
                                   │
                          사내 지식 데이터스토어
                          (사규·매뉴얼·FAQ + 출처)
```

| 종류 | 도구 | 쓰임 | 와우 포인트 |
|---|---|---|---|
| 📚 **데이터스토어** | 사내 지식 (Vertex AI Search) | 사규·매뉴얼·FAQ 검색 | **답변에 인용 출처가 자동으로** 붙음 — 감사·법무가 좋아함 |
| 🔌 **Workspace 커넥터** | Gmail | 메일 조회·초안 생성 | 발송 전 검토 단계 → 위험 0 |
| 🔌 **Workspace 커넥터** | Drive | 파일 읽기·문서 저장 | 산출물이 **내 Drive에 그대로** |
| 🔌 **Workspace 커넥터** | Calendar | 일정 조회 | "내일 미팅" 자동 인식 |
| 🧩 **Custom MCP** | 임직원 디렉토리 | 부서·담당자·이메일·내선 | **원본 DB 노출 없이** 필요한 필드만 |
| 🧩 **Custom MCP** | 거래/고객 DB | 거래처 정보·거래 이력 | 실시간·정밀 조회 (검색 아님) |
| 🧩 **Custom MCP** | 외부 회사 검색 | 외부 회사 개요·뉴스 | 사내 데이터에 **없는 것**을 메움 |

> **적재적소 한 줄** — 문서 검색·출처 인용은 **데이터스토어**, 시스템 조회·실시간·액션은 **MCP**. 에이전트는 필요한 도구만 골라 묶습니다. (오늘 본인이 *조립자*가 되어 봅니다.)

### 커넥터(데이터스토어) vs Custom MCP — 언제 무엇을?

> **한 줄 직관** — 데이터스토어는 **"책에서 찾아 줘"**(문서에 적힌 것), Custom MCP는 **"시스템 두드려 가져와 줘"**(운영 DB 안의 살아있는 값). 같은 일이 아니라 **다른 계층**입니다. 섹션 3-0에서 같은 질문을 두 도구에 던져 직접 대결시켜 볼 거예요.
>
> 자주 나오는 질문: *"BigQuery·Storage에 데이터를 넣는 커넥터가 있는데, 굳이 Custom MCP를 만들 필요가 있나요?"* → 위의 한 줄 직관이 답이에요. 운영 DB의 *현재 값*은 인덱싱으로 풀리지 않습니다.

| | 커넥터 → 데이터스토어 (BigQuery·GCS) | Custom MCP |
|---|---|---|
| 하는 일 | 데이터를 **미리 인덱싱**해 **검색·그라운딩**(읽기) | 내 함수를 **실시간 호출** — 정밀 조회·로직·액션 |
| 잘하는 것 | 문서·자료 의미 검색·요약, **출처 자동 인용** | 키로 정확히 조회, 외부 API, 쓰기/발송, 비즈니스 로직 |
| 데이터 | 내가 미리 넣어둔 내 회사 데이터 | 내 시스템이든 **외부**든, 실시간 |
| 결과 | "관련 있는 내용"(검색 결과) | "정해진 형식의 정확한 값"(함수 반환) |
| 코드 | No-Code(연결만) | 함수 작성·배포 필요 |

**Custom MCP가 꼭 필요한 경우** (아니면 커넥터로 충분):

- **외부 데이터·API** — 외부 회사·뉴스처럼 내 DB에 없는 것. 커넥터로 불가 → MCP가 답.
- **실시간·정밀 조회** — "거래처 X의 최근거래일" 같은 키 기반 정확한 값(검색이 아니라 조회).
- **액션** — 메일 발송·티켓 생성·레코드 수정 등 '읽기'가 아닌 '행동'.
- **외부·레거시 시스템** — 30+ 커넥터에 없는 ERP·사내 마이크로서비스.
- **재사용·거버넌스** — 한 도구를 여러 에이전트가 표준으로 공유, 원본 DB 대신 함수 결과만 노출.

> **결론** — 내 문서·데이터를 **검색·요약**만 하면 → **커넥터(데이터스토어)**가 더 간단(출처 자동). **외부 API·실시간·정밀 조회·액션**이면 → **Custom MCP**. 오늘 *외부 회사 검색*이 바로 커넥터로는 못 해서 MCP가 필요한 대표 사례입니다.

### Custom MCP 작동 점검 — 스모크 테스트

> Workspace(Calendar·Drive·Gmail) 연동 **전**, Custom MCP만 붙인 상태에서 각 도구가 살아있는지 빠르게 확인합니다. Custom MCP를 도구로 추가한 에이전트의 **테스트(파란 창)** 또는 채팅에서 아래를 하나씩 붙여넣어 보세요.

**먼저 — 어떤 도구가 붙어 있나:**

```
지금 네가 사용할 수 있는 도구(MCP) 목록과 각 도구가 하는 일을 알려줘.
```

**① 임직원 디렉토리 MCP** — 정상이면 이름·부서·내선 반환:

```
출장비 정산 담당자가 누구야? 이름·부서·이메일·내선번호를 알려줘.
```

**② 거래/고객 DB MCP** — 정상이면 회사명·거래일·담당자 반환:

```
거래처 'D메디칼' 정보랑 우리 거래 이력(거래시작일·최근거래일)을 알려줘.
```

**③ 외부 회사 검색 MCP** — 도구가 응답하면(찾았든 못 찾았든) 연결 정상:

```
외부 회사 검색 도구로 'D메디칼'(또는 아는 회사명)의 개요를 찾아줘.
```

**④ KPI 대시보드 MCP** *(붙여 두었다면)*:

```
KPI 대시보드에서 이번 분기 매출과 목표 달성률을 보여줘.
```

**판별 팁**

- 답변에 **구체적 데이터(이름·내선·거래일·수치)** 가 나오고 출처/도구 호출 흔적이 보이면 → **MCP 정상**.
- "정보를 찾을 수 없다"거나 막연한 일반 설명만 나오면 → **도구 미연동·미호출**. 에이전트 설정(빨간 창)에서 해당 MCP가 **도구로 추가**됐는지, 지시문이 그 도구를 쓰도록 돼 있는지 확인.
- 첫 호출이 느리거나 실패하면 **콜드 스타트** — 새 세션에서 한 번 더 시도.

> ⚠️ **Calendar·Drive·Gmail은 아직 연동 전**이라 일정·파일·메일 관련 프롬프트(예: "내일 일정 알려줘")는 지금 동작하지 않습니다. Workspace 연동 후 점검하세요.

에이전트 동작: **PLAN → DELEGATE → SYNTHESIZE → RESPOND**

> **왜 이게 'wow'인가** — 보통 AI 과제는 *데이터·도구·시스템 연계*가 안 돼서 PoC에서 멈춥니다(Readiness 부족). 지금 30초 만에 붙이는 Custom MCP는 **그 연계를 미리 풀어둔 것**이에요. 그래서 No-Code로도 **하나의 워크플로우**가 즉시 돕니다 — 단일 작업 자동화가 아니라 **정보수집 → 판단 → 연결·전달**을 *한 흐름에 가로지르는* 진짜 AX입니다.

### 🧭 에이전트 만들기 — 공통 5단계 (한 번만 익히면 끝)

> 앞으로 만들 에이전트 3개 모두 **이 5단계 반복**입니다. 화면 위치를 한 번 기억하세요.

1. 좌측 메뉴 → **+ 새 에이전트** (또는 **Create agent**) 클릭
2. **이름** 입력 → 우측 **빨간 창(설정·Instructions)** 에 역할·절차 붙여넣기
3. 화면 **도구(Tools) 패널** → **+ 도구 추가** → 필요한 커넥터/MCP 체크
4. 좌측 **파란 창(테스트·Test)** 에서 프롬프트 던지고 결과 확인 → 잘 나오면 OK
5. 우측 상단 **저장/생성(Create)** → 좌측 에이전트 목록에 내 에이전트가 뜸

### 3-0. ⚡ Quick Win — 3분 첫 에이전트 (Custom MCP의 진짜 가치)

> **WOW** — "**문서엔 절대 없는, 운영 시스템 안의 실시간 데이터**"를 채팅으로 가져옵니다. 거래 DB·CRM·ERP — 일반 LLM이나 단순 문서 검색이 절대 닿지 못하는 영역이에요. 여기가 Custom MCP의 진짜 차이입니다.

**왜 "거래처 조회"인가** — 사규·매뉴얼 같은 문서는 **데이터스토어**에 이미 잘 들어갑니다(섹션 1에서 봤죠). 하지만 *"D메디칼의 누적 거래액"* 같은 건 어디에도 문서로 적혀있지 않고, **운영 DB 안에서 실시간으로 조회**해야 합니다. → 이게 MCP가 꼭 필요한 순간입니다.

**만들기 (60초)**

### 🆚 데이터스토어 vs Custom MCP — 같은 질문, 다른 답 (5분, 직접 대결)

> 방금 만든 에이전트에 `사내 지식 데이터스토어`만 **추가로** 끼워 보세요(도구 추가 → 사내 지식 체크). 이제 두 도구가 같은 에이전트 안에 있습니다. 아래 두 라운드를 던지고 **에이전트가 어느 도구를 부르는지** 관찰하세요.

**🔵 Round 1 — 정책 질문 (데이터스토어의 영역)**

```
출장비 한도랑 정산 기한 알려줘. 출처도 함께.
```

> **예상 결과** — 사내 지식 데이터스토어 호출
> > 국내 일비 3만 원/일, 해외 미화 80달러/일. 숙박 국내 10만 원/박, 해외 18만 원/박(영수증 실비). **정산은 복귀 후 7영업일 이내** 영수증 첨부 → 경비시스템 제출.
> > *— 출처: 『한빛테크놀로지 사내 정책 — 출장 규정』 §1.2 출장비 일비 / §1.5 정산*
>
> 👉 **데이터스토어가 잘하는 일**: 사규·매뉴얼·FAQ에 **명시적으로 적혀 있는 내용**을 **출처와 함께** 답함. 거래 DB MCP는 호출도 하지 않습니다.

**🟢 Round 2 — 운영 데이터 질문 (MCP의 영역)**

```
거래처 D메디칼 누적 거래액이랑 미수금 얼마야?
```

> **예상 결과** — 거래/고객 DB MCP 호출
> > 누적 거래액 **11.8억**, 미수금 **0원** (결제 정상). 최근거래 2026-03-22 (HanbitOps 2.0 갱신, 4.2억).
>
> 👉 **MCP가 잘하는 일**: 운영 DB의 **실시간·정확한 값**을 가져옴. 같은 질문을 데이터스토어에 던지면 *"관련 문서를 찾을 수 없습니다"* — 거래 정보는 어디에도 문서로 인덱싱돼 있지 않으니까요.

**🧠 한 줄로 기억하기**

| | 데이터스토어 (커넥터로 인덱싱) | Custom MCP |
|---|---|---|
| 닿는 곳 | **문서에 적힌 것** (사규·매뉴얼·FAQ) | **시스템 안의 살아있는 값** (CRM·ERP·내부 API) |
| 답하는 방식 | 의미 검색 + 출처 인용 | 함수 호출 + 정확한 필드 반환 |
| 적합 질문 | "*규칙·절차·정의가 어떻게 돼 있어?*" | "*지금 그 값이 얼마야? 누구야? 언제야?*" |
| 한 줄 | "**책에서 찾아 줘**" | "**시스템 두드려 가져와 줘**" |

> **느낀 점 한 줄** — *"문서 검색은 일반 챗봇도 흉내냅니다. 운영 시스템에서 실시간 값을 가져오는 건 Custom MCP가 있어야만 가능해요."* — 이게 엔터프라이즈 AI의 진짜 분기점입니다. 이제 #1·#2로 가서 둘을 **묶어** 진짜 워크플로우를 만듭니다.

### 3-1. 에이전트 #1 — 🚨 분기 갱신 위험 조기경보 + 대응 플레이북

> **WOW** — 영업본부장이 영업기획팀에 시켜서 **2~3일** 걸리던 분기 갱신 위험 분석을 **30초**에. 단순 정보 조회가 아니라 **위험도 점수 + 우리 영업 플레이북 기반 의사결정 추천 + 7일 액션 플랜**까지. 단일 BI 대시보드나 일반 챗봇으로는 절대 만들 수 없는 합성품.

**왜 이게 진짜 'WOW'인가** — 이 에이전트는 *우리 회사의 판단 기준(영업 플레이북)* 이라는 **데이터스토어** 자산을 *운영 DB의 실시간 데이터(거래 추세·미수금)* 와 *외부 위험 신호(뉴스·공시)* 에 **그대로 적용**해 추천을 만듭니다. 데이터스토어와 MCP가 **분리된 두 도구가 아니라, 합쳐서 판단을 만드는 한 쌍**이 되는 순간 — 이게 엔터프라이즈 AI의 진짜 차이입니다.

**Before / After 한눈에**

| | 지금까지 (Before) | 이 에이전트 후 (After) |

## 4. 정리 & 월요일 액션 플랜 (10분)

### 오늘의 핵심 3줄

1. Gemini Enterprise는 회사 데이터·도구에 연결돼 **실제 일을 하는 'AI 동료'** — 검색 답변이 아니라 *실행 결과*.
2. **문서는 데이터스토어(출처 자동), 도구는 MCP** — 적재적소로 표준화하면 누가 써도 같은 품질.
3. 진짜 가치는 "도입했다"가 아니라 **업무 워크플로우에 녹아드는 것** — 한 명이 만든 에이전트를 **팀 전체가 같은 품질로 사용**.

### 🗓️ 월요일 아침 3단계 — 회사로 가져갈 짐 챙기기

> 오늘 본 그림을 **실제 우리 회사**에서 어떻게 출발시킬지 3단계로 압축했습니다.

| 단계 | 누가 | 무엇을 | 결과물 |
|---|---|---|---|
| **① 도착 첫날** | 본인 | 내 업무 흐름 한 줄 + 가장 답답한 1지점 + 거기 붙일 도구 1개 적기 | **포스트잇 한 장** |
| **② 첫 주 안** | IT/플랫폼팀과 30분 | 그 도구가 **커넥터로 충분**한지 **Custom MCP 필요**한지 결정 | **기술 선택지 1줄** |
| **③ 2주 안** | 본인 + IT | 그 도구를 **하나** 사전 준비 → **No-Code로 1개 에이전트** 만들어 5명에게 공유 | **돌아가는 첫 사내 에이전트** |

> **No-Code의 의미** — 데이터·도구·인증의 *기초공사*는 IT가, 그 위에 얹는 *워크플로우*는 현업이. 이 분업이 가능해진 게 오늘 만난 변화의 핵심입니다.

### 내 워크플로우에서 MCP 후보 찾기 (3분 워크시트)

> 옆 사람과 1분씩 나눠 채워 보세요.

1. **내 업무 흐름 한 줄 (AS-IS)** — 예: "고객 문의 접수 → 분류 → 담당 배정 → 응대"
2. **가장 답답한 지점** (6지점 중 하나) — 정보수집 / 분류·요약 / 판단·우선순위 / 연결·전달 / 고객대응 / 학습·개선
3. **거기 붙일 도구 1개** — 커넥터(검색·요약)면 데이터스토어, 실시간·정밀 조회·액션이면 MCP

> 이 한 줄이 여러분 조직의 **첫 AX 워크플로우 과제**입니다. (좋은 과제는 "AI가 할 수 있는 일"이 아니라 "워크플로우를 바꾸면 성과가 달라지는 지점"에서 나옵니다.)

### 와우 포인트 회고 — 오늘 내가 느낀 순간은?

> 가장 인상 깊었던 순간에 √ 표시해 보세요. 회사 보고서·기획서에서 이 표현이 가장 잘 통합니다.

- [ ] 거래처 조회 에이전트가 **운영 DB의 실시간 누적거래액·미수금**을 표로 줬을 때
- [ ] 같은 에이전트에 정책 질문을 던지자 **데이터스토어로 자동 라우팅**되며 사규 출처가 붙었을 때
- [ ] 갱신 위험 에이전트가 **우리 영업 플레이북 §조항을 인용하며 의사결정 추천**까지 한 줄 안에 묶어 줬을 때
- [ ] 그 결과가 **위험도 점수 + 외부 뉴스 URL + 우리 담당자 + 7일 액션**까지 한 답변에 동시에 나왔을 때
- [ ] 사내 정책 답변에 **사규 조항이 출처로** 자동 인용됐을 때
- [ ] Gmail 초안이 담당자 정보까지 **이미 채워서** 열렸을 때
- [ ] 내가 만든 에이전트를 **공유 링크로 옆자리에 30초 만에** 전달했을 때

**토론 질문**
- 갱신 위험 조기경보 에이전트 — 우리 조직에서 *이 분석을 며칠 들여 하던 사람*은 누구였을까? (영업기획 / 사업개발 / CS / RM)
- 같은 패턴으로 또 무엇을 만들 수 있을까? (예: *신규 거래 신용평가 360°* — 거래DB + 외부뉴스 + 신용평가 가이드라인 데이터스토어)
- 사내 정책 안내 에이전트 — 어디서 가장 답답했나? (인사 / 총무 / IT 헬프데스크)
- "데이터스토어(우리 회사의 판단 기준) × MCP(실시간 데이터)" 한 쌍 — 우리 회사에 또 어떤 *판단 기준 문서*가 있나? (구매 가이드 / 채용 기준 / 보안 정책 / 리스크 매뉴얼)
